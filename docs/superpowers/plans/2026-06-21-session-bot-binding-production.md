# Session Bot Binding Production Plan

## Goal

Turn the current session bot-binding POC into a production-ready WeCom task-binding flow:

- A user can bind an existing CCEM session/task to a WeCom conversation target.
- CCEM can proactively send a task card/update to that WeCom target.
- A quoted WeCom reply can route back into the bound CCEM session.
- The flow reuses WeCom's existing bot-side authorization and recipient configuration. CCEM must not introduce a second user-facing allowlist or a separate bot target picker that duplicates the WeCom bot configuration.
- The implementation must be verified end-to-end with the configured `webot` WeCom bot.

## Product Alignment

### What This Is

This is a session/task binding feature. The user's primary action is:

> "Take this running CC/CCEM session and connect it to this already configured WeCom bot conversation."

After binding, CCEM publishes a small WeCom message that carries enough correlation data for replies to return to the correct session.

### What This Is Not

- Not a new WeCom authorization system.
- Not a second allowlist beside the WeCom bot's own user/group controls.
- Not a global Settings flow.
- Not a generic "remote control targets" administration surface.

Existing WeCom configuration remains in `ChatApp` / `WecomPanel`. Session binding belongs at the task/session edge, with defaults pulled from the selected WeCom bot.

- `apps/desktop/src-tauri/src/channel.rs`
  - `ChannelKind::BotBinding`.
- `apps/desktop/src-tauri/src/main.rs`

The missing production pieces are WeCom delivery, quote/reply routing, UI affordance, and real webot verification.

## Ownership Model

### WeCom Bot Config Owns Authorization

Use existing WeCom fields for access control:

- `adminUserIds`
- `allowedUserIds`
- `allowedGroupChatIds`
- `userAccessPolicy`
- `requireMention`
- permission modes

The bot-binding feature must rely on those checks when handling inbound WeCom messages. Do not add a separate `remoteControlWhitelist` or similar user-visible list.

### CCEM Binding Owns Correlation

Bot binding owns:

- `binding_id`
- `session_id`
- `runtime_id`
- target `bot_id`
- target `peer_id`
- WeCom message/correlation marker
- active/disabled lifecycle

Binding state is about "where should this session send updates and which replies belong to it", not about who is allowed to use WeCom.

### WeCom Defaults Are Convenience, Not Security

Add optional defaults to the existing WeCom bot config:

- `taskBindingDefaultTargetType`: `user` or `group`
- `taskBindingDefaultPeerId`: WeCom user id or group chat id
- `taskBindingAutoSendCard`: boolean, default true

These defaults reduce repeated typing when binding a session. They do not grant permission by themselves.

## UX Shape

### WeCom Panel

Location: `apps/desktop/src/components/chat-app/wecom/WecomPanel.tsx`

Add a compact "Task binding defaults" section near runtime/default environment controls:

- Target type segmented control: User / Group.
- Target id input.
- Auto-send task card toggle.

No long explanatory copy. The structure should make clear this is a default recipient, not an authorization list.

### Session-Level Binding Entry

Add a session/task action that opens a bind dialog:

- Select WeCom bot.
- Show its default target if configured.
- Allow overriding target id for this binding.
- Confirm creates binding and optionally sends the WeCom card.

If the UI already has a session action menu, add the entry there. If not, create the smallest local affordance near existing session/task controls.

### Inbound Reply Model

For MVP, route only messages that clearly reference the bound task:

- Quote/reply to a CCEM-generated WeCom card.
- Or include an explicit binding/task marker generated by CCEM.

Unquoted general WeCom messages continue through the existing headless session flow.

## Backend Tasks

### 1. Config Schema

Files:

- `apps/desktop/src-tauri/src/wecom/types.rs`
- `apps/desktop/src/lib/tauri-ipc.ts`
- `apps/desktop/src/lib/remote-platforms.ts`

Add optional task-binding default fields to the WeCom bot config and make sure serialization stays backward-compatible with existing `~/.ccem/wecom.json`.

Acceptance:

- Old configs load without migration failure.
- Saving a bot preserves the new defaults.
- Frontend types match Rust types.

### 2. Persistent Binding State

File:

- `apps/desktop/src-tauri/src/bot_binding.rs`

Productionize state enough for real app usage:

- Include target bot id and target peer id.
- Include optional WeCom message id/correlation id once delivery succeeds.
- Keep existing request-file ingestion.
- Keep in-memory outbox for diagnostics, but make delivery extensible through a sink.

Acceptance:

- Existing `bot_binding` tests still pass.
- New tests cover binding creation, delivery metadata update, and route lookup by correlation marker.

### 3. WeCom Delivery Hook

Files:

- `apps/desktop/src-tauri/src/wecom/mod.rs`
- `apps/desktop/src-tauri/src/wecom/channel.rs`
- `apps/desktop/src-tauri/src/bot_binding.rs`
- `apps/desktop/src-tauri/src/main.rs`

Add a CCEM-originated delivery path from bot-binding manager to the active WeCom bot connection.

Important implementation note:

- Verify the actual WeCom AI bot SDK command before locking the protocol. Current code only has inbound-response streaming via `aibot_respond_msg`; proactive task-card delivery may need the SDK's `sendMessage(chatid, body)` path rather than a guessed raw command.

Acceptance:

- If the target bot is disconnected, binding creation returns a clear delivery status rather than pretending delivery succeeded.
- If the target bot is connected, CCEM sends a task card/message to the target peer.
- The sent message contains a stable correlation marker.

### 4. WeCom Reply Routing

Files:

- `apps/desktop/src-tauri/src/wecom/mod.rs`
- `apps/desktop/src-tauri/src/wecom/message.rs`
- `apps/desktop/src-tauri/src/bot_binding.rs`

After existing WeCom authorization gates and message normalization, detect quoted/correlated task replies before creating or reusing a normal headless session.

Acceptance:

- Authorized quoted replies are routed into the bound runtime/session.
- Unauthorized actors are still rejected by existing WeCom policy.
- General messages still follow the existing headless session behavior.

## Verification Plan

### Static And Unit

Run at minimum:

```bash
pnpm --filter @ccem/core build
pnpm --filter ccem build
pnpm --filter @ccem/desktop build
cd apps/desktop/src-tauri && cargo test bot_binding
cd apps/desktop/src-tauri && cargo test wecom
```

If backend runtime code changes, also run:

```bash
cd apps/desktop/src-tauri && cargo test native_runtime
```

### Desktop Self-Test

Run:

```bash
pnpm --dir apps/desktop tauri dev
```

Then connect through Tauri MCP and verify:

- WeCom settings include the `webot` bot without exposing secrets.
- A bot-binding request can be created.
- Desktop processes the request.
- Binding appears in `list_session_bot_bindings`.
- If webot is connected, a task card/message is delivered.

### Real Webot Test

With the configured `webot` bot:

- Send a binding card/message from CCEM to a configured target.
- Confirm visible delivery or connection-level success evidence.
- Reply with a quoted/correlated message.
- Confirm the message is routed into the bound session instead of creating a new unrelated headless session.

If the environment cannot complete the real WeCom send, final status must say exactly which leg is blocked: bot connection, SDK command, network, user authorization, or missing target id.

## Subagent Execution Order

Do not run implementation agents in overlapping write sets.

1. Backend worker:
   - Config schema.
   - Binding state.
   - Delivery sink skeleton.
   - Unit tests.
2. Backend worker:
   - WeCom proactive delivery and quoted reply routing.
   - WeCom tests.
3. Frontend/CLI worker:
   - WeCom defaults UI.
   - Session bind dialog.
   - CLI/skill copy cleanup.
4. Main agent:
   - Integration review.
   - Build/test commands.
   - Tauri MCP self-test.
   - Real webot send/reply proof.

Each worker must commit only its own relevant changes using Conventional Commit format.

## Final Acceptance

The goal is complete only when all are true:

- Implementation is present in the worktree.
- Relevant builds/tests pass or failures are documented with exact cause.
- Desktop flow has been exercised through Tauri MCP.
- The webot real-message path has either succeeded with evidence or is blocked by a precise external condition.
- No unrelated dirty-tree edits are reverted or staged.
