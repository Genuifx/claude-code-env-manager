Agent Memory Palace
Agent Memory 模块设计讲解
01 / Framing

记忆宫殿

不是“把聊天记录存起来”，而是给 agent 一套有地址、有层级、有检索成本的记忆系统。

从“向量垃圾场”升级到“可导航的认知建筑”。

4 核心记忆层 Working / Episodic / Semantic / Procedural
2 关键路径 Write path 与 Retrieval path 必须分开设计
1 设计目标 让 agent 先定位房间 再取具体记忆

这份 deck 讲什么

1
为什么“记忆宫殿”这个比喻有用 它强迫你思考地址、作用域、路径和检索代价，而不是只想“我有个 memory store”。
2
外部实践到底在怎么做 Letta、MemGPT、LangGraph、Mem0、Generative Agents、HEBBS 各自抓住了不同维度。
3
如果你要自己做 最小可行架构长什么样 数据模型、写入策略、检索流程、版本与衰减机制都要明确。

02 / Problem

先别急着存
先定义失败模式

多数 agent 的“记忆模块”最后会变成三个问题：写入过多、检索过粗、更新过脏。它看起来像积累知识，实际上是在制造噪音。

失败 1: 全量堆积

把所有对话和工具调用都当成长期记忆，最终上下文里充满低价值回声。

失败 2: 单次相似度检索

永远只做 top-k embedding，缺少作用域和路径，检索结果既贵又不稳定。

失败 3: 只会新增 不会修订

用户偏好变了、代码结构变了、事实过期了，旧记忆还在不断污染决策。

失败 4: 没有记忆权限

个人记忆、项目记忆、组织记忆混在一起，多 agent 场景很快失控。

记忆宫殿带来的第一个转变

旧思路
“把什么东西记下来？”
新思路
“这段记忆属于哪一层、哪个房间、谁能访问、多久过期、检索需要花多大代价？”
Addressable memory
Scoped retrieval
Revision over accumulation
Cost-aware recall
03 / Palace Model

把“宫殿”翻译成工程结构

宫殿不是玄学。它在工程上等价于一组可导航的命名空间：楼层决定范围，房间决定主题，架子决定类型，物件决定单条记忆。

楼层 user / org / project / session
房间 preferences / codebase / incidents / goals
架子 episodic / semantic / procedural / graph
物件 具体 fact / event / insight / edge
先走地址 再展开内容
为什么有用 检索时先缩小范围，减少误召回；写入时先决定归属，减少记忆污染。
为什么不是纯文件夹 同一记忆可以挂多个边：similar_to、derived_from、contradicts、caused_by。
宫殿的关键不是“空间感”

而是“记忆有位置”。位置意味着可摘要、可权限控制、可衰减、可定向检索。

第一层检索要尽量便宜

先用 room summary、key slots、graph anchors 判断值不值得继续往下挖，而不是直接拉原始 episode。

房间本身也要有元数据

比如用途描述、可信度、更新时间、写入策略、最大容量、允许访问的 agent 角色。

04 / Layers

四层记忆
不要混成一锅

长期可用的 agent memory，几乎都会走向分层。区别只在名字不同，结构大体一致。

Working 当前任务 scratchpad，和线程生命周期绑定。
Episodic 原始经历流：事件、对话、工具调用、结果。
Semantic 提炼后的稳定事实、偏好、长期约束。
Procedural 怎么做事：套路、模板、执行风格、工作规约。

每一层的职责

Working
快、脏、短命。只服务当前 rollout，不应该直接进长期 memory。
Episodic
保留可追溯证据。之后做 reflection、debug、audit 都要靠它。
Semantic
真正“下次还该记得”的内容。必须支持 update 和 contradiction。
Procedural
用户偏好的交付方式、团队约定、工具使用策略。它决定行为风格，不只是知识。
Letta、MemGPT、Mem0、Generative Agents 虽然命名不同，但都在做这种层次拆分。
05 / Write Path

记忆写入
比检索更重要

memory 模块的脏数据问题，通常不是检索算法造成的，而是写入时缺少判断。你应该把写入当成一个小型决策系统。

Step 1 从当前 interaction 里抽候选记忆，不要直接全量落库。
Step 2 判定作用域与房间：user、project、session；preferences、codebase、goals、incidents。
Step 3 做 memory action：ADD / UPDATE / DELETE / NONE。
Step 4 为记忆打上证据链：source episode、confidence、valid_until、revision_id。
Step 5 异步 consolidation：把 episodic 压缩成 semantic / procedural insights。 
{
  "candidate": "User prefers terse technical summaries",
  "scope": "user",
  "room": "preferences",
  "layer": "procedural",
  "action": "UPDATE",
  "confidence": 0.88,
  "source_episode_id": "ep_2026_04_02_019d...",
  "revises": "mem_user_pref_summary_style_v3",
  "valid_until": null
}
强约束

长期 memory 写入必须可撤回、可替换、可追溯。否则越用越不敢信。

产品启发

Mem0 把这件事做得最显式：它不是“只会加”，而是把更新规则本身做成模块。

06 / Retrieval

检索不要一步到位
先粗后细

真正像“宫殿”的地方在这里。你不该一上来就取原始记忆，而应该沿路径层层展开。

第一跳 Scope filter: 只看 user / org / project / session 中相关层。
第二跳 Room summary: preferences? codebase? incidents? 先取摘要而不是正文。
第三跳 Graph expansion: 按 derived_from / contradicts / similar_to 展开邻居。
第四跳 Episode fetch: 只把高价值证据拉进 prompt，控制 token 成本。

为什么这样做

A
便宜 先摘要后正文，token 和延迟都更稳定。
B
准 先用地址过滤，减少 embedding-only 的误召回。
C
可解释 你能说清楚这条记忆是从哪个房间、哪条边、哪个证据来的。
CueR 和 HEBBS 对“分保真 recall”“先便宜再深入”这件事给了很强的启发。
07 / Landscape

外网实践
到底各自抓了什么

别把它们当“谁对谁错”。它们更像是在不同位置上切问题：状态管理、长期存储、反思、图结构、开发者体验。

Letta Memory blocks 很像“房间”，有 pinned state、description、shared memory 的概念。 MemGPT 把 memory 当“操作系统里的层级存储”，重点是 context management，不只是存储本身。 LangGraph / Deep Agents 工程上非常实用：把长期记忆和工作文件分开，让 `/memories/` 成为持久层入口。 Mem0 最值得学的是 update/delete 规则和分层 memory types，不是只会 append。 Generative Agents 经典三件套：memory、reflection、planning。说明 episode 需要被提炼成 insight。 HEBBS 少数真的把 palace / graph / contradiction / decay 组合进产品隐喻的方向。
08 / Data Model

一个够用的
Memory Palace schema

你不需要一上来做超复杂数据库，但必须把核心对象定义清楚：memory item、room、edge、revision、policy。

memory_item {
  id,
  scope,          // user | org | project | session
  room,           // preferences | codebase | goals | incidents
  layer,          // episodic | semantic | procedural
  title,
  content,
  summary,
  confidence,
  status,         // active | superseded | expired
  source_episode_id,
  revision_of,
  valid_from,
  valid_until,
  created_at,
  updated_at
}

memory_edge {
  from_id,
  to_id,
  relation,       // derived_from | similar_to | contradicts | caused_by
  weight
}

实现建议

MVP
SQLite 或 Postgres 就够了；图边单表也够用，不必一开始上图数据库。
检索
结构化过滤优先，embedding 作为 room 内二次排序，而不是唯一入口。
版本
不要直接覆盖记忆。用 revision_of 和 status=superseded 保留历史。
清理
session / task 级 episode 自动过期；semantic 只有在被新证据推翻时才收缩。
09 / Build Plan

最小落地路线
不要一口吃成系统

如果你要自己做，最合理的顺序不是“先搞 fancy recall”，而是先把记忆写脏的问题堵住。

建议分三段推进

v0 分层 + 作用域 + ADD/UPDATE/DELETE/NONE。先让 memory 可控。
v1 room summary + graph edges + cost-aware retrieval。开始像“宫殿”。
v2 reflection jobs、contradiction detection、shared rooms、agent-specific policies。
先做什么

把“候选记忆提取器”和“记忆动作决策器”拆出来，先把写入行为模块化。

暂时别做什么

不要一上来就做超级复杂的知识图谱 UI 或多阶段 agent 自主反思调度。

真正的门槛

不是存储，是记忆治理：权限、更新、过期、冲突、证据链、检索预算。

10 / Takeaway

一句话总结

记忆宫殿给你的真正启发，不是“更会记”，而是“更会组织记忆”。

如果你正在做 agent memory 模块，优先追求这五件事：分层、地址化、修订能力、路径式检索、证据与版本。

你可以直接拿去落地的 checklist

1. memory item 必须有 scope / room / layer / revision / source 没有地址与版本，就没有可治理的长期记忆。 2. 写入必须返回 ADD / UPDATE / DELETE / NONE 拒绝“只会 append” 的记忆模块。 3. 检索必须先便宜后昂贵 先 summary，后 episode；先 scope，后 similarity。 4. 每条长期记忆都要能追溯证据 否则 debugging、trust、replay 都会出问题。
