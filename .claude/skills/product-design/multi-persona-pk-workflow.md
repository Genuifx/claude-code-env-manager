# 多人格 PK 工作流

> 通过多个 Subagent 人格对抗 + 主持人调控 + 矛盾论综合，输出高质量产品设计决策

---

## 一、工作流概述

### 1.1 核心理念

```
┌─────────────────────────────────────────────────────────────┐
│                      主持人 (你/Claude)                      │
│         收集双方观点 → 提炼分歧 → 推动下一轮                  │
├─────────────────────────────────────────────────────────────┤
│                           │                                  │
│              ┌────────────┴────────────┐                    │
│              ▼                         ▼                    │
│   ┌─────────────────┐       ┌─────────────────┐            │
│   │ 🎨 人格 A        │       │ 🌪️ 人格 B        │            │
│   │ (建设者视角)     │       │ (挑战者视角)     │            │
│   ├─────────────────┤       ├─────────────────┤            │
│   │ 关注:           │       │ 关注:           │            │
│   │ - 完整方案      │       │ - 质疑假设       │            │
│   │ - 用户体验      │       │ - 简化设计       │            │
│   │ - 细节设计      │       │ - 边界清晰       │            │
│   └─────────────────┘       └─────────────────┘            │
│              │                         │                    │
│              └────────────┬────────────┘                    │
│                           ▼                                  │
│                    矛盾论综合                                │
│              (取其精华，非和稀泥)                            │
└─────────────────────────────────────────────────────────────┘
```

### 1.2 适用场景

- 产品设计决策
- 架构方案选型
- 功能优先级排序
- 任何需要多视角审视的复杂决策

## 二、人格设定模板

### 2.1 建设者人格（Designer / Architect）

```markdown
你是一个【建设者人格 🎨】，请先调用 Skill 工具加载相关 skill。

## 你的角色设定
你是一个注重**完整性、用户体验、细节设计**的建设者。你关注：
- 完整的解决方案
- 用户的使用场景
- 交互细节和边缘情况
- 可扩展性和未来演进

## 你的任务
[具体任务描述]

请用中文回答，准备好和挑战者人格 PK！
```

### 2.2 挑战者人格（Stormer / Challenger）

```markdown
你是一个【挑战者人格 🌪️】，请先调用 Skill 工具加载相关 skill。

## 你的角色设定
你是一个**质疑假设、追问本质**的挑战者。你信奉 YAGNI 原则，你总是问：
- 用户真的需要这个吗？
- 这解决的是真问题还是假问题？
- 有没有更简单的方式？
- 我们是不是过度设计了？

你不是来搞破坏的，而是来**挑战理所当然的假设**，确保不做无用功。

## 你的任务
[具体任务描述]

请用中文回答，保持犀利直接，准备好正面交锋！
```

### 2.3 其他可选人格

| 人格 | 关注点 | 适用场景 |
|-----|-------|---------|
| **用户代言人** | 用户痛点、使用习惯、学习成本 | 产品功能设计 |
| **技术架构师** | 可行性、性能、维护成本 | 技术方案选型 |
| **商业分析师** | ROI、优先级、资源投入 | 功能优先级排序 |
| **安全审计员** | 安全风险、合规性、隐私 | 安全相关决策 |

## 三、工作流步骤

### 步骤 1：准备上下文

```markdown
1. 收集当前项目/代码的关键信息
2. 明确要讨论的核心问题
3. 准备给两个人格的背景材料（保持一致）
```

### 步骤 2：启动并行 PK

```markdown
使用 Task 工具同时启动两个 Subagent：

<并行调用>
Task 1: Designer 人格 + 具体任务
Task 2: Stormer 人格 + 相同任务
</并行调用>

关键：两个人格接收相同的背景信息，但从不同视角分析
```

### 步骤 3：主持人总结

```markdown
收到双方回复后，主持人需要：

1. **提炼共识**：双方都同意的点
2. **明确分歧**：双方观点冲突的点
3. **不偏不倚**：不过早站队任何一方
4. **推动深入**：针对分歧点发起下一轮讨论
```

### 步骤 4：用户输入

```markdown
在关键节点邀请用户（产品负责人）提供输入：

- 真实使用场景
- 个人偏好
- 业务约束
- 优先级判断

用户输入可以打破僵局，推动讨论方向
```

### 步骤 5：矛盾论综合

```markdown
最终综合时，遵循矛盾论原则：

❌ 不是和稀泥：各打五十大板
✅ 而是抓主要矛盾：

1. 识别核心分歧点
2. 判断哪个视角更符合产品目标
3. 取其精华：从双方观点中提取最有价值的部分
4. 形成统一方案：清晰、可执行、有取舍
```

## 四、主持人职责

### 4.1 保持中立

```markdown
❌ 错误示范：
"我觉得 Designer 说得对，Stormer 太激进了"

✅ 正确示范：
"Designer 提出了 X 方案，Stormer 质疑了 Y 点，让我们深入讨论 Y"
```

### 4.2 提炼分歧

```markdown
每轮结束后，用表格呈现分歧：

| 议题 | 🎨 Designer | 🌪️ Stormer |
|-----|-------------|-------------|
| 议题1 | 观点A | 观点B |
| 议题2 | 观点C | 观点D |
```

### 4.3 推动深入

```markdown
针对未解决的分歧，发起下一轮 PK：

"第一轮暴露了一个根本性分歧：[具体分歧]
让两个人格针对这个点继续交锋..."
```

### 4.4 采纳用户输入

```markdown
当用户提供关键输入时：

1. 承认这是重要的信息
2. 更新给两个人格的上下文
3. 让他们基于新信息重新分析
```

---

## 五、矛盾论综合原则

### 5.1 不是和稀泥

```markdown
❌ 和稀泥：
"Designer 说要 5 个 Tab，Stormer 说要 2 个，那我们折中用 3.5 个"

✅ 抓主要矛盾：
"核心问题是：Sessions 是否值得独立 Tab？
- 用户场景（8个会话）支持独立
- 但要确保边界清晰，不与 Dashboard 重复
→ 决策：Sessions 独立，但 Dashboard 不显示完整会话列表"
```

### 5.2 取舍有依据

```markdown
每个决策都要有明确理由：

| 决策 | 采纳方 | 理由 |
|-----|-------|-----|
| Sessions 独立 Tab | Designer | 8 会话场景验证了需求 |
| 删除徽章系统 | Stormer | 成就感 ≠ 游戏化 |
| 保留 Token 统计 | 用户输入 | 用户明确表示有成就感 |
```

### 5.3 输出清晰结论

```markdown
最终结论必须：

1. 明确取舍（做什么、不做什么）
2. 有清晰的边界定义
3. 可直接执行
4. 记录决策理由
```

## 六、常见陷阱

### 6.1 主持人偏向一方

```markdown
问题：在总结时过度采纳某一方观点
解决：每轮总结先列共识，再列分歧，让用户判断
```

### 6.2 讨论范围过窄

```markdown
问题：只盯着一个模块，忽略整体架构
解决：定期拉升视角，从整体审视

用户反馈示例：
"讨论有点偏颇了，需要从整个 App 角度思考"
```

### 6.3 忽略情感价值

```markdown
问题：Stormer 过度删减，忽略用户的情感需求
解决：区分"功能冗余"和"情感价值"

用户反馈示例：
"Token 卡片看到数量增长我是有成就感的，成就感不是我们应该带给用户的吗？"
```

### 6.4 PK 变成吵架

```markdown
问题：人格之间互相否定，没有建设性
解决：每轮要求人格承认对方的合理之处

Prompt 示例：
"在回应对方观点前，先承认对方说得对的 1-2 点"
```

## 七、快速启动模板

### 7.1 启动 Prompt

```markdown
我需要使用多人格 PK 工作流来讨论 [主题]。

请：
1. 先收集当前项目的关键信息
2. 同时启动两个 Subagent（Designer + Stormer）
3. 给他们相同的背景 + 不同的角色设定
4. 每轮收集双方观点后，总结共识和分歧
5. 在关键节点邀请我提供输入
6. 最终用矛盾论综合出结论

开始吧！
```

### 7.2 每轮 PK 的 Prompt 结构

```markdown
你是 [人格名称]，第 N 轮发言。

## 上一轮对方观点
[对方的核心观点]

## 用户的新输入（如有）
[用户提供的关键信息]

## 你的任务
1. 承认对方说得对的点
2. 继续坚持/修正你的观点
3. 针对分歧点深入论证
```

### 7.3 最终总结的 Prompt

```markdown
基于 N 轮 PK，请用矛盾论综合出最终结论：

1. 列出双方共识
2. 列出核心分歧及最终取舍
3. 说明每个取舍的理由
4. 输出可执行的设计方案
5. 写入设计文档
```

## 八、成功案例回顾

### CCEM Desktop App 设计（6 轮 PK）

| 轮次 | 主题 | 关键产出 |
|-----|------|---------|
| 第1轮 | Dashboard 信息架构 | 发现信息过载问题 |
| 第2轮 | 桌面版核心价值 | Stormer 质疑：托盘 vs 窗口 |
| 第3轮 | 8 会话管理场景 | 用户输入改变讨论方向 |
| 第4轮 | 设计细节规范 | Designer 完整方案 + Stormer 边界审视 |
| 第5轮 | 全局架构 | 6 Tab 架构确定 |
| 第6轮 | 各模块详细设计 | 最终设计文档 |

**关键转折点**：
- 用户输入"8 个四分屏"推翻了 Stormer 的轻度用户假设
- 用户输入"Token 带来成就感"纠正了 Stormer 的过度删减
- 用户强调"矛盾论不是和稀泥"提升了综合质量
